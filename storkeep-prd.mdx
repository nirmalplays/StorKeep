## 1. The Problem

### 1.1 Filecoin Foundation's Failing KPI

[Filecoin Foundation's public 2025 KR2](https://fil.org/blog/filecoin-foundation-s-vision-and-strategic-priorities-for-2025):

> **"75% of on-chain deals are renewed before expiring"**

This is a publicly stated, measurable goal. The chain data shows it is failing:

| Quarter | Active Storage | QoQ Change | Root Cause |
|---------|---------------|-----------|------------|
| Q4 2024 | 1,500 PiB | — | Baseline |
| Q1 2025 | 1,300 PiB | **−13%** | Deal expirations outpaced new onboarding |
| Q2 2025 | 1,100 PiB | **−11%** | "Expirations of prior large contracts removed more capacity than was replaced" |
| Q3 2025 | 1,110 PiB | −1% | Stabilised on high-value verified enterprise data |

Source: [Messari State of Filecoin Q1](https://messari.io/report/state-of-filecoin-q1-2025) · [Q2](https://messari.io/report/state-of-filecoin-q2-2025) · [Q3 2025](https://messari.io/report/state-of-filecoin-q3-2025)

The Q2 divergence is the clearest proof: [daily new storage deals rose 25% QoQ while total active storage fell 11%](https://messari.io/report/state-of-filecoin-q2-2025). Deals expire faster than they are renewed. This is not a prediction — it is the chain telling you in real numbers.

### 1.2 Why Deals Don't Get Renewed

There are three compounding reasons:

**Reason 1 — No autopilot exists for average developers.**
The [RaaS (Replication, Renewal and Repair) primitive](https://docs.filecoin.io/smart-contracts/programmatic-storage/raas) was designed for this in FVM, but [RaaS nodes must be self-hosted](https://docs.filecoin.io/smart-contracts/programmatic-storage/raas) and only monitor deals originated through aggregators. Self-hosting means almost nobody runs it.

**Reason 2 — Lighthouse's hosted RaaS has fixed global params.**
[Lighthouse deployed universal RaaS nodes](https://docs.lighthouse.storage/lighthouse-1/filecoin-first/section-b) on mainnet (`0xd928b92E6028463910b2005d118C2edE16C38a2a`) and Calibration (`0x4015c3E5453d38Df71539C0F7440603C69784d7a`). But the renewal parameters are set globally — you cannot customise thresholds per deal, and it only works for Lighthouse-aggregated CIDs. From [their docs](https://docs.lighthouse.storage/lighthouse-1/filecoin-first/section-b): *"The params for renewal and repair have been decided by lighthouse universally for all deals."*

**Reason 3 — Payment is a manual human step.**
Even when renewal is triggered, someone must pay in FIL. AI agents, dApps, and automated pipelines — the primary future clients of Filecoin storage — cannot click a MetaMask popup. There is no way to pay for deal renewal programmatically without building custom infrastructure.

**Reason 4 — No developer SDK exists.**
There is no npm package exposing `renewDeal(dealId)`. Developers who want autopilot storage today must understand RaaS contracts, Lighthouse aggregation, FVM smart contract interactions, and Filecoin JSON-RPC — all at once.

### 1.3 The F5 Gap

[F5 at ETHGlobal](https://ethglobal.com/showcase/f5-p34yq) was a hackathon deal-renewal smart contract. It identified the right problem but had no payment layer, no SDK, no hosted service, no npm package. It was never adopted. StorKeep is what F5 should have become.

### 1.4 The FilOz Signal

At [FDS-7 Buenos Aires (November 2025)](https://filecoin.io/blog/posts/behind-the-scenes-in-buenos-aires-), the Filecoin core team (FilOz) was actively designing **"x402 on Filecoin Pay + Pay-to-Access Retrievals"** — a session presented by Jennifer Wang (FilOz) and Miroslav Bajtoš (Space Meridian). You are building what the core team is prototyping. That is exactly where to be at a hackathon.

## 2. Why x402

### 2.1 What x402 Is

[x402](https://github.com/coinbase/x402) (5,400 ⭐ · Apache-2.0 · 193 contributors) is Coinbase and Cloudflare's open protocol for HTTP-native autonomous payments. [Launched May 2025, it has processed over 100 million transactions](https://www.x402.org/writing/x402-v2-launch). It revives the dormant HTTP 402 status code to make USDC payments as simple as an HTTP header.

The complete payment flow in one diagram:

```
AI Agent / cron job / dApp
       │
       ├─── POST /api/deals/5847291/renew ──────────────────▶ StorKeep API
       │◀── 402 Payment Required ────────────────────────────
       │    PAYMENT-REQUIRED: { amount: "0.25 USDC",
       │                        network: "eip155:84532",   ← Base Sepolia
       │                        payTo:   "0xStorKeep..." }
       │
       │    [x402 client in SDK signs EIP-3009 USDC authorization]
       │    [No MetaMask. No popup. No human.]
       │
       ├─── POST /api/deals/5847291/renew ──────────────────▶ StorKeep API
       │    PAYMENT-SIGNATURE: <signed x402 payload>          │
       │                                                       ├── Coinbase CDP facilitator verifies on Base
       │                                                       ├── Calls Lighthouse RaaS contract on FVM
       │                                                       ├── Emits DealRenewalTriggered on Filecoin chain
       │◀── 200 OK ────────────────────────────────────────────
            { renewed: true, txHash: "0xFilecoin...",
              paymentTxHash: "0xBase...",
              newExpiryEpoch: 2847291 }
```

### 2.3 The Exact Client-Side Code (Verified from npm docs)

The `@x402/fetch` package exposes `wrapFetchWithPaymentFromConfig`. This is what StorKeep SDK uses internally — the developer never sees it:

```typescript
// Verified from @x402/fetch npm package docs
import { wrapFetchWithPaymentFromConfig } from '@x402/fetch'
import { ExactEvmScheme } from '@x402/evm'
import { privateKeyToAccount } from 'viem/accounts'

const account = privateKeyToAccount(process.env.EVM_PRIVATE_KEY as `0x${string}`)

const fetchWithPayment = wrapFetchWithPaymentFromConfig(fetch, {
  schemes: [
    {
      network: 'eip155:84532',  // Base Sepolia
      client: new ExactEvmScheme(account),
    },
  ],
})

// From here: any fetch() that returns 402 is handled automatically
// Payment is signed and request is retried — zero extra code
const response = await fetchWithPayment('https://api.storkeep.xyz/api/deals/5847291/renew', {
  method: 'POST',
})
```

### 2.5 Why x402 Is Perfect Here

The deal renewal problem is fundamentally an **autonomous payment problem**. The client is an AI agent, a GitHub Action, a cron job, or a dApp — none of which can click a MetaMask popup.

[Coinbase's Agentic Wallets (February 2026)](https://www.coinbase.com/developer-platform/discover/launches/agentic-wallets) provide purpose-built infrastructure for exactly this: AI agents that autonomously hold USDC and pay for services via x402. [Coinbase AgentKit](https://github.com/coinbase/agentkit) gives any AI agent a wallet that works natively with x402.

```typescript
// An AI agent that never loses Filecoin storage
import { AgentKit } from '@coinbase/agentkit'
import { StorKeep } from 'storkeep-sdk'

const agentKit = await AgentKit.from({
  cdpApiKeyId: process.env.CDP_KEY_ID,
  cdpApiKeySecret: process.env.CDP_KEY_SECRET,
})

const sk = new StorKeep({ x402Wallet: agentKit.wallet, network: 'calibration' })

await sk.enableAutopilot({
  dealId: '5847291',
  renewWhenEpochsLeft: 100_000,  // ~35 days warning
  maxPriceUsdc: 1.00,
})

// Agent wallet is charged $0.25 USDC on Base when renewal triggers.
// No human approved this. This is the entire point.
```

## 3. What StorKeep Is

StorKeep has three components:

```
┌─────────────────────────────────────────────────────────────┐
│                    storkeep-sdk (npm)                        │
│  getDealStatus() · renewDeal() · enableAutopilot()           │
│  Uses @x402/fetch internally — developer sees nothing        │
├─────────────────────────────────────────────────────────────┤
│              StorKeep Backend (x402-gated REST API)          │
│  Hosted on Railway · x402 middleware via @x402/express       │
│  Reads Filecoin chain · Calls Lighthouse RaaS · Emits events │
├──────────────────────┬──────────────────────────────────────┤
│  Filecoin FVM        │  Base (Sepolia / Mainnet)             │
│  StorKeepRegistry    │  USDC payments via x402               │
│  .sol contract       │  Coinbase CDP facilitator             │
│  Renewal audit log   │  $0.25 per renewal                    │
└──────────────────────┴──────────────────────────────────────┘
```

**Component A: `storkeep-sdk` on npm**
The TypeScript package developers install. Three public methods. Handles x402 payments internally. Built with [tsup](https://tsup.egoist.dev/) — CJS + ESM + `.d.ts`.

**Component B: StorKeep Backend**
An Express server gated by `@x402/express` middleware. Monitors Filecoin deals via [Glif RPC](https://lotus.filecoin.io/lotus/developers/glif-nodes/), triggers renewals through the [Lighthouse RaaS smart contract](https://docs.lighthouse.storage/lighthouse-1/filecoin-first/section-b), and records every renewal on-chain.

**Component C: StorKeepRegistry.sol**
A minimal FVM contract that emits `DealRenewalTriggered` events — a public, verifiable audit trail of every renewal StorKeep has ever performed.

### 5.2 `getDealStatus(dealId)` — Free, No Payment

Queries the Filecoin chain directly via [Glif JSON-RPC](https://lotus.filecoin.io/lotus/developers/glif-nodes/) using `Filecoin.StateMarketStorageDeal`. No x402 payment. No SDK token needed.

```typescript
const status = await sk.getDealStatus('5847291')
```

**Returns `DealStatus`:**

```typescript
interface DealStatus {
  dealId:              string
  pieceCid:            string           // bafyreid...
  clientAddress:       string           // f410f... Filecoin address
  providerMinerId:     string           // f012345
  startEpoch:          number
  endEpoch:            number
  currentEpoch:        number           // fetched from Filecoin.ChainHead
  epochsUntilExpiry:   number           // endEpoch - currentEpoch
  daysUntilExpiry:     number           // epochsUntilExpiry / 2880
  needsRenewal:        boolean          // true if < renewThreshold
  renewalCostUsdc:     string           // "$0.25" — from StorKeep API
  status:              'active' | 'expiring' | 'expired'
}
```

**Example output:**

```
{
  dealId: '5847291',
  pieceCid: 'bafyreidcr2b4tbqlvlh...',
  clientAddress: 'f410fxrp4y...',
  providerMinerId: 'f012345',
  startEpoch: 1284920,
  endEpoch: 2132211,
  currentEpoch: 2120000,
  epochsUntilExpiry: 12211,
  daysUntilExpiry: 4.2,
  needsRenewal: true,
  renewalCostUsdc: '0.25',
  status: 'expiring'

```

**How it works internally:**

```typescript
// Filecoin JSON-RPC call — verified from Ankr docs + Glif docs
const response = await fetch('https://api.calibration.node.glif.io/rpc/v1', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    jsonrpc: '2.0',

const { result } = await response.json()
// result.Proposal.StartEpoch, result.Proposal.EndEpoch, result.Proposal.PieceCID['/']
```

### 5.3 `renewDeal(dealId, options?)` — x402 Payment Automatic

Triggers a single renewal. The SDK's internal x402 client intercepts the 402 response, signs the USDC payment from the configured wallet, and retries — all transparently.

```typescript
const result = await sk.renewDeal('5847291', {
  maxPriceUsdc:    1.00,        // refuse if cost > $1.00 USDC
  durationEpochs:  518_400,     // ~6 months — MIN_DEAL_DURATION_EPOCHS
})
```

**Returns `RenewalResult`:**

```typescript
interface RenewalResult {
  renewed:         boolean
  dealId:          string
  txHash:          string    // Filecoin chain transaction hash
  paymentTxHash:   string    // Base chain USDC payment transaction hash
  actualCostUsdc:  string    // "$0.25"
  newExpiryEpoch:  number
  newExpiryDate:   string    // ISO-8601
  filfoxUrl:       string    // https://calibration.filfox.info/en/tx/0x...
  basescanUrl:     string    // https://sepolia.basescan.org/tx/0x...
  lighthouseJobId: string    // Lighthouse RaaS job reference
}
```

**Options type:**

```typescript
interface RenewOptions {
  maxPriceUsdc?:   number   // default: 1.00
  durationEpochs?: number   // default: 518_400 (min deal duration, ~6 months)
}
```

**Throws:**

- `PriceExceededError` — if `actualCostUsdc > maxPriceUsdc`
- `DealNotFoundError` — if `dealId` does not exist on-chain
- `DealExpiredError` — if deal already expired (past `endEpoch`)
- `InsufficientUsdcError` — if wallet USDC balance < renewal cost
- `X402PaymentError` — if x402 facilitator rejects the payment
- `RenewalFailedError` — if Lighthouse RaaS contract reverts

### 5.4 `enableAutopilot(config)` — Set and Forget

Registers the deal with StorKeep's monitoring service. The cron job (runs every 6 hours) polls chain state and triggers renewal automatically when `epochsUntilExpiry < renewWhenEpochsLeft`. Payment is deducted from the wallet passed to the `StorKeep` constructor — no approval needed.

```typescript
const autopilot = await sk.enableAutopilot({
  dealId:               '5847291',
  renewWhenEpochsLeft:  100_000,    // ~35 days before expiry
  maxPriceUsdc:         1.00,       // per-renewal budget cap
  webhookUrl:           'https://myapp.com/storkeep-events',  // optional
  webhookSecret:        'sk_whsec_...',                        // optional HMAC
})
```

**Returns `AutopilotRegistration`:**

### 5.8 Error Classes

All extend `StorKeepError extends Error`. Import directly:

```typescript
import { StorKeep, PriceExceededError, DealExpiredError } from 'storkeep-sdk'

try {
  await sk.renewDeal('5847291', { maxPriceUsdc: 0.10 })
} catch (e) {
  if (e instanceof PriceExceededError) {
    console.log(`Too expensive: $${e.actualCostUsdc} > $${e.maxPriceUsdc}`)
  }
  if (e instanceof DealExpiredError) {
    console.log('Deal already expired — cannot renew')
  }
}
```

**Full error hierarchy:**

```typescript
class StorKeepError extends Error {
  code: string
  details?: unknown
}

// Payment errors
class InsufficientUsdcError  extends StorKeepError {} // wallet USDC < renewal cost
class PriceExceededError     extends StorKeepError {  // cost > maxPriceUsdc
  actualCostUsdc: string
  maxPriceUsdc:   string
}
class X402PaymentError       extends StorKeepError {} // CDP facilitator rejected

// Filecoin errors
class DealNotFoundError      extends StorKeepError {} // dealId not on chain
class DealExpiredError       extends StorKeepError {} // past endEpoch
class RenewalFailedError     extends StorKeepError {  // Lighthouse RaaS reverted
  lighthouseError: string
}

// Config errors
```

## 6. The Backend Service

### 6.1 Routes

```
GET  /api/deals/:dealId/status      Free · queries Filecoin RPC
POST /api/deals/:dealId/renew       x402-gated $0.25 · triggers Lighthouse RaaS
GET  /api/autopilot/:dealId         Free · returns config + history
POST /api/autopilot                 x402-gated $0.10 · registers monitoring
DELETE /api/autopilot/:dealId       Free · stops monitoring
GET  /api/health                    Free · health check
GET  /api/stats                     Free · total renewals, deals monitored
```

### 6.2 The `POST /api/deals/:dealId/renew` Flow

When called with a valid x402 USDC payment:

1. **Verify deal exists** — `Filecoin.StateMarketStorageDeal` via Glif RPC. Extract `PieceCID` and `EndEpoch`.
2. **Check deal is not expired** — if `currentEpoch > endEpoch`, return `DealExpiredError`.
3. **Verify PieceCID is on IPFS** — Lighthouse RaaS requires the CID to be pinned. Check `https://api.lighthouse.storage/api/lighthouse/get_proof?cid=<cid>`.
4. **Submit to Lighthouse RaaS** — call `submitRaaS(pieceCid, replications=1, repairThreshold=28800, renewThreshold=240)` on the Lighthouse contract. Contract address from [Lighthouse RaaS docs](https://docs.lighthouse.storage/lighthouse-1/filecoin-first/section-b).
5. **Record on FVM** — call `StorKeepRegistry.recordRenewal(dealId, pieceCid, payerAddress, lighthouseJobId)`.
6. **Return result** — include `txHash` (FVM), `paymentTxHash` (Base), `filfoxUrl`, `basescanUrl`.

## 7. The FVM Smart Contract

### 7.1 `StorKeepRegistry.sol`

A minimal on-chain audit log. Every renewal StorKeep triggers is recorded here — providing verifiable proof that is publicly readable by anyone.

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.17;

/// @title StorKeep Registry — On-chain renewal audit trail for Filecoin deals
/// @notice Records every renewal triggered via StorKeep for public verifiability
/// @dev Deployed on Filecoin Calibration testnet. Verified on Filfox.
contract StorKeepRegistry {

    struct Renewal {
        uint64  dealId;
        bytes   pieceCid;
        address triggeredBy;      // address that paid via x402
        uint256 timestamp;
        string  lighthouseJobId;
    }

    mapping(uint64 => Renewal[]) public renewalHistory;
    uint256 public totalRenewals;

    event DealRenewalTriggered(
        uint64  indexed dealId,
        bytes           pieceCid,
        address indexed triggeredBy,
        string          lighthouseJobId,
        uint256         timestamp
    );

    /// @notice Called by StorKeep backend after successful Lighthouse RaaS submission
    function recordRenewal(
        uint64  dealId,
        bytes   calldata pieceCid,
        address triggeredBy,
        string  calldata lighthouseJobId
    ) external {
        renewalHistory[dealId].push(Renewal({
            dealId:          dealId,
            pieceCid:        pieceCid,
            triggeredBy:     triggeredBy,
            timestamp:       block.timestamp,
            lighthouseJobId: lighthouseJobId
        }));
        totalRenewals++;
        emit DealRenewalTriggered(dealId, pieceCid, triggeredBy, lighthouseJobId, block.timestamp);
    }

    /// @notice Returns full renewal history for a deal
    function getRenewalHistory(uint64 dealId) external view returns (Renewal[] memory) {
        return renewalHistory[dealId];
    }

    /// @notice Total renewals ever processed by StorKeep
    function getTotalRenewals() external view returns (uint256) {
        return totalRenewals;
    }
}
```

## 8. x402 Integration Deep Dive

### 8.1 How the SDK Internally Wraps x402

Inside `src/x402/client.ts`:

```typescript
// src/x402/client.ts — internal, not exported publicly
import { wrapFetchWithPaymentFromConfig } from '@x402/fetch'
import { ExactEvmScheme } from '@x402/evm'
import type { WalletClient } from 'viem'

export function createX402Fetch(wallet: WalletClient, network: string) {
  const account = wallet.account!

  return wrapFetchWithPaymentFromConfig(fetch, {
    schemes: [
      {
        network,                           // 'eip155:84532' for Base Sepolia
        client: new ExactEvmScheme(account),
      },
    ],
  })
}
```

Inside `StorKeep.ts`, the class stores this wrapped fetch and uses it for all API calls:

```typescript
// src/StorKeep.ts (simplified)
export class StorKeep {
  private readonly x402Fetch: ReturnType<typeof wrapFetchWithPaymentFromConfig>
  private readonly config:    NetworkConfig

  constructor(options: StorKeepConfig) {
    this.config     = getNetworkConfig(options.network ?? 'calibration')
    this.x402Fetch  = createX402Fetch(options.x402Wallet, this.config.x402Network)
  }

  async renewDeal(dealId: string, opts: RenewOptions = {}): Promise<RenewalResult> {
    // This fetch call will automatically handle 402 if it occurs
    const response = await this.x402Fetch(
      `${this.config.storkeepApiUrl}/api/deals/${dealId}/renew`,
      {
        method:  'POST',
        headers: { 'Content-Type': 'application/json' },
        body:    JSON.stringify({ durationEpochs: opts.durationEpochs ?? MIN_DEAL_DURATION }),
      }
    )

    if (!response.ok) {
      const error = await response.json()
      this.throwTypedError(error)
    }

    return response.json() as Promise<RenewalResult>
  }
}
```

```typescript
export const MIN_DEAL_DURATION = 518_400     // 6 months minimum
export const MAX_DEAL_DURATION = 1_555_200   // 18 months maximum

export function epochsToMs(epochs: number): number {
  return epochs * 30_000
}

export function msToEpochs(ms: number): number {
  return Math.floor(ms / 30_000)
}

export function daysToEpochs(days: number): number {
  return Math.round(days * EPOCHS_PER_DAY)
}

export function epochsToHuman(epochs: number): string {
  const totalSeconds = epochs * 30
  const days    = Math.floor(totalSeconds / 86400)
  const hours   = Math.floor((totalSeconds % 86400) / 3600)
  const minutes = Math.floor((totalSeconds % 3600) / 60)

  if (days > 0) return `${days}d ${hours}h`
  if (hours > 0) return `${hours}h ${minutes}m`
  return `${minutes}m`
}
```
